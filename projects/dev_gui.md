# GUI/TUIアプリケーションの設計・実装要件

アプリケーションがGUIまたはTUIを持つ場合、以下の要件に従ってアーキテクチャを設計し、実装すること。

## 0. 新規実装時の基本方針（重要）
GUI/TUIアプリケーションを新規実装する場合、まずは自前で厳密なメインループやFPS制御を構築するのではなく、対象環境におけるデファクトスタンダードな既存ツールの標準機能を優先して利用し、短時間かつバグの少ない実装を提案・構築すること。

* **Python (Windows / Raspberry Pi / UserLAnd 等):** 
  * GUIの場合は Pygame などを利用し、`pygame.time.Clock().tick(FPS)` によるスリープ・遅延吸収、イベントキューのフラッシュ、`pygame.display.flip()` によるダブルバッファリングに依存すること。
  * TUIの場合は Textual などの非同期イベントループと再描画エンジンを利用すること。
* **C++ (M5Paper / ESP32 等):** 
  * M5Unified や LovyanGFX のスプライト機能（メモリ上の仮想キャンバス）を利用し、裏バッファへの描画と `pushSprite()` による実画面への転送を行うこと。

※ 上記のような既存フレームワークの機能に依存せず、あえて独自にメインループを実装する必要がある場合、またはフレームワークのラッパーを自作する場合は、以下の「独自メインループ設計要件」に必ず従うこと。

## 1. 独自メインループ設計要件 (FPS設定と状態管理)
必ず目標FPS（15, 30, 60, 120のいずれか）を設定できる設計にすること。フレームの更新は以下の手順で厳密に制御すること。

1. **初期化処理**
   - フレームカウンタ（`FC`）とテンポラリフレームカウンタ（`tFC`）を用意し、初期化する。
   - ループ開始時の実時刻を `BaseClocktime` として取得する。

2. **目標時刻の算出**
   - 次回フレーム時刻を `BaseClocktime + (tFC / FPS)` として算出する。

3. **入力処理**
   - フレームの先頭で入力バッファをフラッシュする。
   - その時点でバッファの最後に入力されたものを「確定入力」とし、フレーム処理中に発生した入力は破棄（または次回に持ち越し）すること。

4. **フレーム構成処理（高負荷処理）**
   - 画面の要素を算出または取得する。
   - 非同期処理で要素を構築している場合は、フレームごとの「要素保持コンテキスト」を用意すること。メモリに余裕がない環境では、この構成処理の直前に全てのフレーム要素をラッチ（コピー）して状態を確定させること。

5. **フレーム描画処理**
   - 構成した要素を裏バッファに描画する（可能であればダブルバッファリングを利用すること）。

6. **スリープと時刻超過時の補正**
   - 処理完了後の現在時刻と次回フレーム時刻を比較する。
   - 余裕時間がある場合は、次回フレーム時刻までスリープする。
   - 処理落ちなどにより現在時刻が次回フレーム時刻を超過している場合は、現在時刻を新たな `BaseClocktime` に更新し、`tFC` を `0` にリセットすること。

7. **実表示（フリップ）**
   - 描画済みのフレームを画面に実表示する（可能であればダブルバッファリングによるフリップ操作を行うこと）。

8. **カウンタの更新**
   - `FC` および `tFC` をインクリメントし、次フレームの処理へ移行する。
